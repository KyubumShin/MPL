# v0.6.7 — Cluster Ralph: Feature-Scoped Verify-Fix Loop

> **Codename**: Cluster Ralph
> **Theme**: "1M 시대의 검증 — 아끼지 말고 다 하자"
> **Date**: 2026-03-24
> **Status**: Planned
> **Prerequisite**: v0.6.6 (B-04 Integration Checkpoints)

---

## Summary

v0.6.7 replaces the mechanical checkpoint system (B-04: every 3 phases) with **feature-aligned cluster verification**. Each cluster of related phases gets its own E2E scenarios and fix loop, ensuring issues are caught and fixed while context is fresh.

Additionally includes 3 quality-of-life improvements: Lint Gate, TSConfig Strictness, and Config Schema consolidation.

### Design Principles (1M Context Era)

- **No token optimization** — 1M context means ~220K pipeline usage (22%). Plenty of headroom.
- **Verify everything** — No skip, no defer, no risk-based exclusion. Every cluster gets full E2E.
- **Fix immediately** — Don't batch fixes. Fix while the code is fresh in context.
- **2-Layer, not 4-Layer** — Phase Runner micro-fix + Cluster Ralph + Final E2E. That's it.

## V-01: Cluster Ralph (Feature-Scoped Verify-Fix Loop)

### Concept

B-04 checkpoint (mechanical, every 3 phases) evolves into **Cluster Ralph** (semantic, per-feature-group):

```
Before (B-04):
  P1 → P2 → P3 → [Checkpoint] → P4 → P5 → [Checkpoint] → ... → 5-Gate
                   generic                    generic              final

After (Cluster Ralph):
  ┌─ Cluster A ──────────────────────┐
  │ P1 → P2 → P3 → [E2E-A + Fix]   │
  └──────────────────────────────────┘
  ┌─ Cluster B ──────────────────────┐
  │ P4 → P5 → [E2E-B + Fix]        │
  └──────────────────────────────────┘
  ┌─ Cluster C ──────────────────────┐
  │ P6 → P7 → [E2E-C + Fix]        │
  └──────────────────────────────────┘
           ↓
  [Final Cross-Feature E2E] → [5-Gate]
```

### Decomposer Output: Cluster Definition

Decomposer already outputs phases with `feature_priority` (CORE/EXTENSION/SUPPORT) and dependency ordering. v0.6.7 adds explicit `clusters` field:

```yaml
clusters:
  - id: cluster-1
    name: "Core CRUD"
    pp_link: ["PP-1", "PP-2"]
    feature_priority: CORE
    phases: ["phase-1", "phase-2", "phase-3"]
    feature_e2e:
      - id: e2e-1a
        scenario: "Create project → create chapter → edit → save → reopen → verify"
        type: integration    # integration | smoke | contract
        commands:
          - "npm test -- --grep 'project crud'"
          - "cargo test project"
      - id: e2e-1b
        scenario: "Delete project → verify cascade deletion of chapters"
        type: integration
        commands:
          - "npm test -- --grep 'cascade'"
      - id: e2e-1c
        scenario: "Dev server starts without crash after all CRUD changes"
        type: smoke
        commands:
          - "timeout 10 npm run dev 2>&1 | head -20"

  - id: cluster-2
    name: "Worldbuilding"
    pp_link: ["PP-3"]
    feature_priority: EXTENSION
    phases: ["phase-4", "phase-5"]
    feature_e2e:
      - id: e2e-2a
        scenario: "Create character → create location → link to chapter → verify reference"
        type: integration
        commands:
          - "npm test -- --grep 'worldbuilding'"

final_e2e:
  - id: final-1
    scenario: "Full user journey: project → chapters → worldbuilding → AI generation → export"
    type: integration
    commands:
      - "npm run test:e2e"
  - id: final-2
    scenario: "Full build succeeds with all changes"
    type: smoke
    commands:
      - "npm run build && cargo build"
```

### Decomposer Clustering Rules

```
Rule 1: Group phases by PP linkage.
  Phases implementing the same PP(s) belong to the same cluster.

Rule 2: Respect dependency order within cluster.
  Phases within a cluster maintain their topological order.

Rule 3: CORE clusters come first (T-12 Core-First Ordering).
  CORE clusters → EXTENSION clusters → SUPPORT clusters.

Rule 4: Max cluster size = 5 phases.
  If a PP spans more than 5 phases, split into sub-clusters at natural boundaries.

Rule 5: Min cluster size = 1 phase.
  Single-phase clusters are valid (e.g., infrastructure scaffold).

Rule 6: Generate 3-5 E2E scenarios per cluster.
  At least 1 integration + 1 smoke scenario per cluster.
  E2E scenarios must be executable (commands field, not descriptions).

Rule 7: Generate final_e2e for cross-feature interactions.
  At least 1 full-journey scenario + 1 full-build smoke.
```

### Orchestrator: Cluster Execution Loop

**Modified Step 4 in `mpl-run-execute.md`:**

```
Step 4.0: Parse clusters from decomposition.yaml

for each cluster in clusters:

  # Step 4.0.1: Announce cluster start
  announce: "[MPL] Starting Cluster '{cluster.name}' ({cluster.phases.length} phases)"

  # Step 4.1-4.3: Execute phases within cluster (existing flow)
  for each phase in cluster.phases:
    Step 4.0.5: JIT Phase Seed Generation
    Step 4.1: Context Assembly
    Step 4.2: Phase Runner Execution
    Step 4.2.2: Test Agent (mandatory domains)
    Step 4.3: Result Processing
    Step 4.3.6: Session Context Persistence
    # Budget check between phases (existing)

  # Step 4.4: Cluster E2E Verification (NEW)
  announce: "[MPL] Cluster '{cluster.name}' — running feature E2E ({cluster.feature_e2e.length} scenarios)"

  cluster_results = []
  for each scenario in cluster.feature_e2e:
    result = execute_scenario(scenario)
    cluster_results.append(result)

  if all_pass(cluster_results):
    announce: "[MPL] Cluster '{cluster.name}' E2E PASSED ✓"
    update RUNBOOK: cluster status = PASS
    continue to next cluster

  else:
    # Cluster Ralph: Fix Loop
    announce: "[MPL] Cluster '{cluster.name}' E2E FAILED — entering Cluster Fix Loop"

    for attempt in 1..2:
      # Step 4.4.1: Failure Analysis
      failed_scenarios = [s for s in cluster_results if s.status == FAIL]

      # Step 4.4.2: Scout Dispatch (root cause analysis)
      scout_result = Task(subagent_type="mpl:mpl-scout", model="haiku",
        prompt="Analyze cluster E2E failures: {failed_scenarios}.
                Identify which phase and which files caused each failure.
                Cluster phases: {cluster.phases}.
                Check the actual code, not just descriptions.")

      # Step 4.4.3: Targeted Fix
      for each identified_issue in scout_result.issues:
        Task(subagent_type="mpl:mpl-phase-runner", model="sonnet",
          prompt="Fix the following issue in {identified_issue.phase}:
                  Problem: {identified_issue.description}
                  Files: {identified_issue.files}
                  Expected: {identified_issue.expected}
                  Scope: ONLY modify files within this cluster's phases.")

      # Step 4.4.4: Re-verify
      cluster_results = []
      for each scenario in cluster.feature_e2e:
        result = execute_scenario(scenario)
        cluster_results.append(result)

      if all_pass(cluster_results):
        announce: "[MPL] Cluster '{cluster.name}' E2E PASSED after fix attempt {attempt} ✓"
        update RUNBOOK: cluster status = PASS (fixed)
        break

      announce: "[MPL] Cluster '{cluster.name}' fix attempt {attempt} — still failing"

    if not all_pass(cluster_results):
      # After max 2 fix attempts, defer and continue
      announce: "[MPL] Cluster '{cluster.name}' E2E deferred after 2 fix attempts"
      update RUNBOOK: cluster status = DEFERRED
      record deferred failures for Final E2E
      continue to next cluster

# Step 4.5: Final Cross-Feature E2E (NEW — replaces part of old 5-Gate)
announce: "[MPL] Running Final Cross-Feature E2E"
for each scenario in final_e2e:
  result = execute_scenario(scenario)
  if FAIL: enter existing 5-Gate Fix Loop

# Step 4.6: 5-Gate Quality (existing, unchanged)
# Gate 0.5, 1, 1.5, 1.7, 2, 3
```

### execute_scenario Helper

```
function execute_scenario(scenario):
  results = []
  for each command in scenario.commands:
    result = Bash(command, timeout=30000)
    results.append({
      command: command,
      exit_code: result.exit_code,
      stdout: result.stdout[:500],  # truncate for context
      stderr: result.stderr[:500]
    })

  return {
    scenario_id: scenario.id,
    scenario_name: scenario.scenario,
    status: all(r.exit_code == 0 for r in results) ? PASS : FAIL,
    details: results
  }
```

### RUNBOOK Updates

After each cluster E2E, append to RUNBOOK:

```markdown
## Cluster: {cluster.name} ({cluster.id})
- Phases: {cluster.phases}
- PP Link: {cluster.pp_link}
- E2E Scenarios: {count} total
- Result: PASS | PASS (fixed, attempt N) | DEFERRED
- Failed Scenarios: [list if any]
- Fix History: [attempt 1: ..., attempt 2: ...]
```

### B-04 Migration

B-04 checkpoint concept is **fully replaced** by Cluster Ralph:

| B-04 Concept | Cluster Ralph Equivalent |
|---|---|
| `checkpoint: true` flag | Implicit — every cluster boundary |
| `verifies_phases: [...]` | `cluster.phases: [...]` |
| `integration_tests: [...]` | `cluster.feature_e2e: [...]` |
| Mechanical insertion (every 3) | Semantic grouping (by PP) |
| No fix loop | Fix loop (max 2) |

**Backward compatibility**: If Decomposer outputs old `checkpoint` format, orchestrator treats it as a single-phase cluster with the checkpoint's `integration_tests` as `feature_e2e`.

### Tier Applicability

| Tier | Cluster Behavior |
|------|-----------------|
| **Frugal** | Single cluster (1 phase) — Cluster E2E = existing Gate 1 behavior |
| **Standard** | Single cluster (1-3 phases) — Cluster E2E adds feature scenarios |
| **Frontier** | Multiple clusters — Full Cluster Ralph with fix loops |

## V-02: Lint Gate (Gate 0.5 Extension)

### Problem

Gate 0.5 currently runs only `tsc --noEmit` / `cargo check`. No lint execution (eslint, ruff, flake8) despite `lint_error` being referenced as a pattern tag.

### Design

Add lint auto-detection and execution to Gate 0.5, **before** type checking:

```
Gate 0.5 Enhanced:
  Step 1: Detect lint tools
    if exists("package.json"):
      pkg = Read(package.json)
      if "eslint" in pkg.devDependencies or exists(".eslintrc*") or exists("eslint.config.*"):
        lint_commands.push("npx eslint . --max-warnings=0")

    if exists("pyproject.toml") or exists("ruff.toml"):
      lint_commands.push("ruff check .")
    elif exists(".flake8") or exists("setup.cfg"):  # flake8 config
      lint_commands.push("flake8 .")

    if exists("biome.json"):
      lint_commands.push("npx biome check .")

  Step 2: Execute lint
    for each cmd in lint_commands:
      result = Bash(cmd, timeout=60000)
      if result.exit_code != 0:
        lint_failures.append({tool, errors: result.stderr})

  Step 3: Existing type check (tsc, cargo check, etc.)

  Step 4: Report
    if lint_failures:
      Gate 0.5 = FAIL (lint)
      → Fix Loop targets lint errors first
```

## V-05: Scope Drift Report (Post-Execution)

### Problem

No mechanism to detect when implementation drifts from declared scope.

### Design

Add drift measurement to Step 5.1 (Final Verification) in Finalize:

```
Step 5.1.5: Scope Drift Report (NEW)

  # Compare declared scope vs actual changes
  declared_files = collect all impact.create + impact.modify from decomposition.yaml
  actual_files = Bash("git diff --name-only HEAD~{commit_count}")

  added_files = actual_files - declared_files    # files changed but not declared
  missed_files = declared_files - actual_files   # files declared but not changed

  drift_ratio = len(added_files) / max(len(declared_files), 1)

  # Report in RUNBOOK
  append to RUNBOOK:
    ## Scope Drift Report
    - Declared files: {len(declared_files)}
    - Actual files: {len(actual_files)}
    - Undeclared changes: {len(added_files)} ({drift_ratio:.0%} drift)
    - Unimplemented declared: {len(missed_files)}
    - Files: [list]

  # No blocking — informational only for v0.6.7
  # Data collection for future Gate integration
```

## Migration (0.6.6 → 0.6.7)

### Breaking Changes: NONE

| Component | Backward Compatible? | Detail |
|-----------|:---:|--------|
| decomposition.yaml | ✅ | `clusters` field is additive; old `checkpoint` format falls back to single-phase cluster |
| Phase Runner | ✅ | No changes to Phase Runner — cluster loop is orchestrator-level |

| Config | ✅ | New `cluster_ralph.max_fix_attempts` field, defaults to 2 |
| Frugal/Standard | ✅ | Single-cluster behavior matches pre-0.6.7 |

### Rollback

Set `cluster_ralph: { enabled: false }` in config.json → orchestrator skips Step 4.4, falls back to direct 5-Gate after all phases.
