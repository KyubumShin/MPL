## Problem Statement

> MPL의 검증 파이프라인은 orchestrator self-report에 의존하며, 이는 구조적으로 거짓을 발행할 수 있다. gate_results는 초기화만 되고 아무도 쓰지 않으며, finalize는 자연어 채널에서 상태를 추출한다. 동시에 orchestrator는 핵심 단계(chain derivation, test-agent dispatch, Phase 0 agent dispatch)를 silent skip하는 패턴이 3회 연속 실험에서 관측됐다.

## Decision

### Layer 1: Hook 기반 실측 강제 (Enforcement)

#### 1.1 gate-recorder.mjs (신규 hook)

PostToolUse on Bash — phase-runner의 gate 명령 실행 후 `state.json.gate_results[name]` 자동 기록.

```javascript
gate_results[gate_name] = {
  command: String,           // 실행된 명령
  exit_code: Number,         // 0 or non-zero
  stdout_tail: String,       // 마지막 500자
  timestamp: ISO String
}
```

**매칭 전략** (#5 설계 수정 반영):
- **Primary**: `.mpl/verify.sh` 존재 시 해당 스크립트의 exit code 기록 (orchestrator 무관)
- **Fallback**: common pattern heuristic (`pnpm test/lint/build`, `cargo test/clippy`) 자동 감지
- **Best-effort**: `state.json.verification_commands[]` (Phase 0 인터뷰 수집 시에만)

**추가 기록**:
- `test_agent_dispatched[phase_id]`: PostToolUse on Task|Agent, subagent_type=mpl-test-agent 감지 시
- `sprint_status.completed_todos`: phase dir count 기반 실시간 increment (finalize 대기 금지)

**불변식**: `gate_results[name] == null || exit_code != 0` → finalize 자동 FAIL.

#### 1.2 Finalize 채널 전환

`commands/mpl-run-finalize.md` line 397의 RUNBOOK.md 자연어 파싱 **완전 제거**.

Before: `gate_results = Read(".mpl/mpl/RUNBOOK.md") → extract "3H+1A Gate Quality Results" section`
After: `gate_results = state.json.gate_results` (gate-recorder hook이 기록한 machine evidence만)

- `gate_results[name] == null` → "NOT EVALUATED" 표기
- `exit_code != 0` → "PARTIAL (exit {N}, {stdout_tail})" 표기
- `exit_code == 0` → "PASS (exit 0, command: {cmd})" 표기
- "✅ clean" 표기는 **모든 gate가 exit 0일 때만** 허용

#### 1.3 phase-controller chain-seed guard

`hooks/mpl-phase-controller.mjs`에 phase 전환 시 chain-seed 활성화 검증:

```javascript
if (config.chain_seed?.enabled === true && !existsSync('.mpl/mpl/chain-assignment.yaml')) {
  console.error('[MPL] ⚠️ chain_seed.enabled=true but chain-assignment.yaml NOT generated.');
}
```

Silent skip을 observable failure로 전환. chain derivation 자체의 hook 강제는 v0.16.0.

### Layer 2: Anti-rationalization 방어 (Persuasion, #44)

agent-skills 패턴(Common Rationalizations + Red Flags)을 4개 커맨드 파일에 추가. Hook 강제가 커버하지 못하는 prompt-level skip 패턴을 정조준.

| 파일 | 방어 대상 | 핵심 변명 → 반박 |
|---|---|---|
| `mpl-run-decompose.md` Step 3-G | chain derivation skip | "Stage 1이니까 optional" → config에 enabled=true면 skip 금지 |
| `mpl-run-execute.md` Step 4.3 | test-agent dispatch skip | "phase-runner가 이미 테스트 작성" → code author = test author는 tautology |
| `mpl-run-finalize.md` Gate 집계 | self-report 의존 | "runner가 passed 보고" → exit code 없으면 pass 모름 |
| `mpl-run-phase0-analysis.md` Step 2 | Phase 0 agent inline skip | "이미 분석했으니 agent 불필요" → 독립 관점이 목적 |

Anti-rationalization은 Hook과 **직교** — 둘 다 적용하여 "설득 + 강제" 이중 방어.

## Rollback Criteria

- finalize 소요 시간 >200% 증가 AND 원인이 gate-recorder hook 로직 → hook 비활성화 (config gate)
- gate-recorder false positive (환경 차이로 인한 실측 실패) >20% → heuristic 매칭 완화
- Anti-rationalization이 orchestrator를 infinite loop에 빠뜨림 → 해당 섹션 제거

## Alternatives Considered

### A: Prompt-only 수정 (Anti-rationalization만, hook 없음)
agent-skills 방식: 모든 규제를 프롬프트로. 비용 최소, 강제력 0. exp9/exp10에서 프롬프트 지시가 무시되는 패턴 반복 관측 → **기각**. 단, Layer 2로 병용은 채택.

### B: 전면 hook 강제 (chain derivation, Phase 0 dispatch, test-agent dispatch 모두)
모든 비활성 agent의 dispatch를 hook으로 강제. 비용 과대 (~200줄+ JS), chain derivation 알고리즘 JS 구현 필요, 한 릴리스에 담기엔 blast radius 과도 → **v0.16.0으로 이월**. v0.15.0은 gate_results 채널 정비에 집중.

### C: verify.sh 의무화 (프로젝트가 미제공 시 파이프라인 차단)
Hard guard: `.mpl/verify.sh` 없으면 gate 불가. 기존 프로젝트 호환성 파괴, 채택 장벽 증가 → **기각**. Heuristic fallback으로 점진적 도입.
