## Context

Commit `4903a8d` (2026-04-10 18:50 KST, *"chore: bump to v0.12.2 — docs consistency + stale agent cleanup"*) deleted two agent files:

- `agents/mpl-ambiguity-resolver.md` (803 lines)
- `agents/mpl-test-agent.md` (155 lines)

The commit message framed the deletion as "stale agent cleanup" and additionally claimed the commit "updated all deleted agent references across 8 doc files." A session audit on 2026-04-11 — and a targeted runtime probe on the same day — surfaced four problems with that framing:

1. The two deletions had **very different actual situations**. `mpl-ambiguity-resolver` was validly migrated to the `mpl_score_ambiguity` MCP tool; `mpl-test-agent` had no replacement at all.
2. The commit message's "updated all references" claim was **false**. Multiple active protocol files retained dispatch references to the deleted agents that the commit did not touch.
3. There is **no decision document** for the v0.12.2 deletion anywhere in `docs/decisions/`, `wiki/pages/decisions/`, or `~/project/decision/`. The only consolidation decision doc (`wiki/pages/decisions/agent-consolidation-16-to-8.md`) predates this commit by five days and does not list either agent as a deletion target.
4. The `wiki/daily/2026-04-10.md` report, written the same day, states "MPL working tree clean, no activity today" — failing to capture `4903a8d` entirely.

The pattern matches an earlier Claude session running a "cleanup pass" without verifying that the "unused" agents were actually unused. This ADR records the forensic evidence, ratifies the ambiguity-resolver deletion as valid, and specifies a partial revert for `mpl-test-agent` as the immediate protective action — while leaving the long-term architectural decision (which agent layer owns adversarial verification) explicitly deferred to the upcoming reconciliation audit.

### Evidence chain

**1. No decision document exists for v0.12.2 deletion.** Exhaustive search across `docs/decisions/` (1 doc — AD-0001 only), `wiki/pages/decisions/` (`agent-consolidation-16-to-8.md` dated 2026-04-05), and `~/project/decision/` (20+ files) produces zero matches for `mpl-test-agent` or `mpl-ambiguity-resolver` as deletion targets in the context of v0.12.2. The 2026-04-05 consolidation doc explicitly lists the 7 agents deleted at `a60320e` (v0.12.1) and does not extend its scope to v0.12.2.

**2. Commit message misrepresents scope.** `4903a8d` touched 16 files. 9 are doc files, 1 is the skill checklist, 2 are the deleted agents, 4 are version-bump artifacts (marketplace.json, plugin.json, READMEs). **None** of the following files — which contain live dispatch references to the deleted agents — were in the commit:

```
commands/mpl-run-execute.md          — L109, L470, L500, L507, L511, L535
commands/mpl-run-execute-gates.md    — L148, L154, L160
commands/mpl-run-phase0.md           — L485, L488 (ambiguity-resolver dispatch, legacy)
hooks/mpl-sentinel-s3.mjs            — L5-6 (header comment references Test Agent output)
hooks/mpl-phase-controller.mjs       — L130, L141-142, L150-151, L181 (legacy hints)
hooks/mpl-ambiguity-gate.mjs         — L6, L82, L94 (legacy hints)
hooks/mpl-validate-output.mjs        — L35, L59 (output validation schema)
agents/mpl-interviewer.md            — L19, L41, L126 (handoff references)
```

The commit message's "Update all deleted agent references across 8 doc files" counted doc-side references only. Protocol and hook-side dispatch references were left dangling. The ambiguity-resolver half of this dangling set was subsequently cleaned up in commit `a4ef45d` (2026-04-12) because its runtime behavior was already correct via the MCP tool — the hint strings were merely misleading. The test-agent half could not be cleaned up the same way because there is no migrated runtime behavior to point the hints at.

**3. Runtime probe confirms test-agent dispatch is hard-broken.** A runtime probe on 2026-04-11 called `Task(subagent_type="mpl-test-agent", …)` and observed:

```
Agent type 'mpl-test-agent' not found. Available agents:
  general-purpose, statusline-setup, Explore, Plan, claude-code-guide,
  codex:codex-rescue, mpl:mpl-codebase-analyzer, mpl:mpl-decomposer,
  mpl:mpl-doctor, mpl:mpl-git-master, mpl:mpl-interviewer,
  mpl:mpl-phase-runner, mpl:mpl-phase0-analyzer
```

This is a **hard error** — not a silent skip, not a fallback to general-purpose. Any MPL run that executes the F-40 Mandatory Independent Verification dispatch at `commands/mpl-run-execute.md:511` for a mandatory-domain phase (ui, api, algorithm, db, ai) will surface this error. The fact that the error has not been observed in the wild since `4903a8d` indicates either that no MPL run has exercised the F-40 path on main, or that the orchestrator has been silently swallowing the error. Either way, the F-40 protocol step is **not running** even though the command doc claims it is.

**4. Empirical weight of what was lost.** The cb-testbed Phase A1 ablation (n=3 + n=5, run 2026-04-09/10) measured cross-boundary defect defense across three mechanisms. Two findings bear directly on this ADR:

- **M2 contract presence is the dominant defense** (Δ = -2.17) — catches most L1 protocol-level defects.
- **L2/L3 test blindness is structural**: C2 = 0 (parameter-level detection) and C3 = 0 (schema-level detection) across all 29 runs. Tests written by the implementer cannot detect missing L2 parameters or missing L3 schema fields, no matter how many tests are added. This is mathematical, not cultural.

The 28 lines of HA-02 Adversarial Verification content inside `agents/mpl-test-agent.md` — 5 Korean self-rationalization anti-patterns, structured Test/Expected/Actual/Verdict output format, Probing Hints integration directive, failure-mode entries — were the only mechanism in MPL that structurally targeted L2/L3 blind spots. When `4903a8d` deleted the file, the defense evaporated. The empirical motivation for that defense was only measured 2026-04-09/10, which is **after** the 2026-04-05 consolidation decision doc but **before** `4903a8d` — the author of `4903a8d` had no way of knowing they were deleting an empirically-validated defense layer.

**5. Ambiguity-resolver deletion is separately valid.** `mcp-server/src/tools/scoring.ts` implements a full replacement:

- 5-dimension scoring (spec_completeness, edge_case_coverage, technical_decision, acceptance_testability, pp_conformance) via LLM API with weighted sum
- Weakest-dimension detection with deterministic tiebreaker
- Suggested-question generation per dimension via template map
- Outputs `{ambiguity_score, clarity_pct, threshold_met, dimensions, weakest_dimension, weakest_dimension_key, weakest_score, suggested_question}`

The orchestrator drives the Stage 2 Socratic loop inline: call the MCP tool, read `suggested_question`, ask the user, append response, re-call the tool, loop until `threshold_met == true`. `hooks/mpl-ambiguity-gate.mjs` enforces the threshold at `Task(mpl-decomposer)` PreToolUse. The runtime loop has been functional since `mpl_score_ambiguity` landed in MCP, and commit `a4ef45d` (2026-04-12) updated the protocol + hook hint strings to match runtime reality. **Issue #6** (closed earlier) verified the gate functions end-to-end.

**6. Test-agent has no replacement and no obvious migration path.** `mcp-server/src/tools/` contains exactly two tool files: `scoring.ts` and `state.ts`. There is no independent-verification tool. Building one is not merely unimplemented — it's architecturally infeasible. F-40's core value is *independent context*: a separate session, spec-only inputs, no view of the runner's transcript. MCP tools run inside the caller's session and cannot provide the independence property. Folding the verification role into `phase-runner` destroys the independence that HA-02's anti-rationalization prompts were designed for. These same constraints were analyzed in AD-0001 S1 for a related injection context.

## Decision

MPL will execute a **partial revert of `4903a8d`** with the following specific actions:

1. **Restore `agents/mpl-test-agent.md`** to its pre-`4903a8d` state via targeted `git checkout 4903a8d^ -- agents/mpl-test-agent.md`. This brings the 155-line agent file (including the 28 lines of HA-02 adversarial verification content) back into the tree and unblocks the F-40 dispatch path.

2. **Keep `agents/mpl-ambiguity-resolver.md` deleted.** The MCP migration is valid, the runtime loop works, and commit `a4ef45d` has already cleaned up the legacy dispatch hint strings. No further action.

3. **Keep all doc updates from `4903a8d`.** The 9 doc files and 4 version-bump artifacts that `4903a8d` touched contain valid changes that remain correct. Full revert is explicitly rejected (see Alternatives).

4. **Defer the long-term architectural path** for how adversarial verification is hosted in MPL (separate agent vs inline role in phase-runner vs full 3-role triangulation redesign vs flat structure compensated by mechanical Hard 3) to the upcoming 4-dimension reconciliation audit (`~/project/wiki/scratch/2026-04-12/mpl-reconciliation-audit.md` §6). The restored baseline is a *working* starting point: the audit will decide whether to keep it, inline it, redesign it, or drop it in favor of mechanical gates, but whatever the audit decides will execute against a working tree — not against git archaeology.

Rationale: the partial revert is **optionality-preserving**. All four downstream architectural options can be executed on top of a restored baseline. Leaving test-agent deleted until the audit would force the audit to reason about a known-broken system for the entire audit + redesign window, and would force the eventual implementation to reconstruct HA-02's 28 lines from git archaeology rather than restore them by checkout — introducing drift risk for zero benefit.

This ADR is an evidence record plus an immediate protective action. It is **not** an architectural decision about how to host adversarial verification in the long run. That decision is explicitly deferred.

## Alternatives Considered

### Full revert of `4903a8d`

**Rejected because**: the commit contained legitimate doc updates (agent count 8→7 in `design.md`, version bump to 0.12.2, skill checklist addition for `docs/config-schema.md`, README_ko consistency pass) that are still correct and wanted. Full revert would also restore `mpl-ambiguity-resolver.md` even though its function is validly migrated to MCP, reintroducing a confused dual-implementation state. The partial revert at the file level — one file, one git checkout — captures the protective action without undoing the valid cleanup.

### Keep both deletions, redesign in audit

**Rejected because**: leaves F-40 actively broken for the entire audit + redesign window (at minimum several days, potentially longer if the architectural debate goes back and forth). The 28 lines of HA-02 content would have to be reconstructed from git history rather than restored by checkout, introducing content drift risk during reconstruction. The audit's architectural debate also benefits from having the original design checked in as a live artifact — treating test-agent as "deleted and lost" biases the debate toward "redesign from scratch" rather than "restore, refactor, or redistribute."

### Inline HA-02 into `mpl-phase-runner` now

**Rejected because**: this pre-empts the architectural debate. Audit §6 option B is precisely "simulate adversarial inside phase-runner via explicit role-switching prompts." Committing to that option before the audit assesses whether same-session role-switching preserves any of the independence property that HA-02 was designed for would lock in an answer before the question is asked. The cb-phase-a1 C2/C3=0 finding suggests that any adversarial verification happening in the same session as the implementation may still lose L2/L3 detection power — inlining now would be a structural regression disguised as a restoration.

### Redesign Test Agent as an MCP tool

**Rejected because**: the core value of F-40 is *independent context* — a separate session, spec-only inputs, no view of the runner's transcript. This is an LLM reasoning task that requires a fresh conversation, not a deterministic function. MCP tools run inside the caller's session and cannot provide the independence property. The same constraint was argued in AD-0001 S1 for a different injection context and remains valid here.

## Consequences

### Immediate

- `agents/mpl-test-agent.md` is restored (155 lines, including 28 lines of HA-02 content) via targeted git checkout.
- F-40 Mandatory Independent Verification becomes dispatchable again. The runtime probe must be re-run post-restore to confirm the hard error is replaced by a successful dispatch.
- Dispatch references at `commands/mpl-run-execute.md:109, 470, 500, 507, 511, 535`, `commands/mpl-run-execute-gates.md:148, 154, 160`, and `hooks/mpl-sentinel-s3.mjs:5-6` are **no longer dangling**. The protocol resumes working as originally intended.
- `agents/mpl-interviewer.md:19, 41, 126` and `hooks/mpl-validate-output.mjs:35, 59` become consistent with the tree again (they reference the agent by name — the restored agent matches the name).
- Agent count returns to 8. `docs/design.md` §Agent Registry and `wiki/pages/projects/mpl.md` header need a follow-up correction, tracked as part of the revert commit or a small follow-up doc touch.

### Deferred to audit

The long-term architectural path for adversarial verification is explicitly deferred. The audit will decide among (at least) these options:

- **(A) Keep `mpl-test-agent.md` as a separate agent.** Restore F-40 as-designed. Agent count stays at 8. Cost: extra orchestration layer, but preserves the independence property.
- **(B) Inline HA-02 adversarial prompt into `mpl-phase-runner` Step 4** with explicit role-switching. Agent count stays at 7. Cost: loses session independence, relies on prompt rigor, open question whether this actually catches L2/L3 defects in practice.
- **(C) Full redesign: 3-role adversarial triangulation (Runner → Adversarial Tester → Reviewer).** Potentially with new agent architecture. Cost: rebuilding orchestration; unknown complexity.
- **(D) Flat structure + compensate via mechanical Hard 3 (AD-05, #19).** Rely on contract + key-set + value-path verification. Cost: gives up adversarial independence entirely; leans on the hypothesis that mechanical parameter-level checking closes the L2 gap fully.

If the audit chooses (B), (C), or (D), the F-40 dispatch references listed in §Evidence will need to be rewritten to match the new architecture. That cleanup is tracked in a follow-up issue and is **not** part of this ADR's scope. If the audit chooses (A), no further action is needed beyond the partial revert.

### Explicitly not decided

- The long-term architecture of MPL's Phase internal verification layer.
- Whether `agents/mpl-test-agent.md` content should be updated to reflect the cb-phase-a1 empirical motivation for HA-02. The restored file will be the exact pre-`4903a8d` content; any content update is a separate commit with a separate justification.
- Whether `docs/design.md` §Agent Registry should stabilize on 7 or 8 agents long-term. This depends entirely on the audit outcome.

### Operational notes

- Restore commit must use `git checkout 4903a8d^ -- agents/mpl-test-agent.md`, **not** a full revert. Touches exactly one file.
- Commit message should reference this ADR by ID (`AD-0003`).
- Post-restore, re-run the runtime probe `Task(subagent_type="mpl-test-agent", …)` to confirm the hard error is gone and the dispatch succeeds. Record the post-revert probe result in the audit doc.
- `wiki/pages/projects/mpl.md` header drift (already says "8 agents" with a 7-row table) becomes self-consistent after the restore, though the list inside the wiki page will still need an explicit update to include `mpl-test-agent`.
