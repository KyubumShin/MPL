04

흐름 시각화

4.1 · Phase Lifecycle — End-to-End

계획 → 실행(phase 루프) → 최종 감사. 각 phase는 fresh context의 executor + 별도 verifier로 분리되어 author-independence를 유지.

4.2 · 단일 Tool-Call이 엔진을 지나가는 시퀀스

각 모듈은 100% 독립 — block 반환 시 나머지 모듈은 실행조차 안 됨. 순서는 dispatch에 명시적으로 선언되어 디버깅 가능.

4.3 · 모듈 의존성 그래프

L2 정책 모듈끼리는 import 금지 (audit만 예외). 모두 L1만 의존 → 현재의 "hook들이 서로 모름" 강점을 모듈 레벨에서 보존.

07

병렬성 + 합의 — 재설계 위에 얹는 층

출발점: MPL은 이미 parallel scheduler가 있지만 prompt 안에만 산다. commands/mpl-run-execute.md Step 4.0의 pseudocode를 orchestrator LLM이 손으로 실행 — 모델이 그 분기를 skip하면 무음 실패. Telemetry/schema는 v7에 다 있지만 dispatcher 코드는 0줄. PR #230이 failure_code를 별도 축으로 분리한 이유.

통합 설계: prompt-only 스케줄러를 deterministic Node-side pipeline으로 교체. Continuous-frontier dispatcher + worktree-per-slot 격리 + per-phase state shard + manifest 기반 4-bucket reconcile, 모두 ExecutionContext tuple로 묶음.

7.1 · Scheduler — Barrier vs Continuous-Frontier

parallel: false 는 tier_mutex_{N} 가상 lock으로 보존 — phase3-gate 같은 단독 phase는 여전히 단독 실행
P4가 P3만 의존하면 P2 종료를 기다릴 필요 없음. Tier 경계가 아닌 reconcile 경계가 유일한 sync.

7.2 · 4-Bucket 합의 — Manifest Diff 결정 트리

합의는 deterministic 4-bucket 분류 + Bucket C만 LLM. N-verifier 다수결 없음 — 단일 verifier가 결정 못 하면 그게 곧 recompose 신호 (planner under-spec).

7.3 · State Shard — Per-Phase 쓰기 + Single Reducer

Worker는 자기 phase의 shard만 쓰고, 단일 reducer가 결정론적으로 병합. Silent last-write-wins은 field-ownership matrix가 차단 — 미선언 필드는 throw.

7.4 · 배포 — 2단계

Stage A · MUST · 4–6주

엔진 + L2 정책 모듈 재설계 머지 후 시작

  • policy/scheduler.mjs — continuous-frontier + heartbeat/watchdog + tier_mutex + resume/pivot
  • policy/isolation.mjs — worktree-per-slot + contract freezing + HIGH-risk rejection
  • state/wave-reducer.mjs + shard-writer.mjs + field-ownership matrix
  • policy/reconcile/ — 4-bucket diff (T/S/C/X)
  • mpl-engine.mjs route_to_phase 프론트도어 + ExecutionContext threading
  • source-edit · channel-registry · permit · contracts · audit → execution_context_id 키 사용
  • hooks/mpl-require-reconciliation.mjs + 6개 신규 failure_code
  • commands/mpl-run-execute.md Step 4.0 pseudocode 삭제

Stage B · NICE · post-v1

Stage A 안정화 후 점진

  • Tree-sitter 심볼 추출 (regex 대체)
  • Priority inheritance — anti-starvation
  • Post-merge consensus recheck — silent 3-way merge 캐치
  • Per-wave hard1_baseline intermediate gate + file-set-hash 캐시
  • exports_symbols_manual_overrides[] 탈출구
3개 kill switch — emergency rollback 보장: scheduler.dispatch_model: barrier (legacy tier-barrier 흐름) · reconcile.enabled: false (오늘의 prose 아티팩트 fallback) · state.writer_mode: single (shard 비활성화, 강제 직렬)
왜 ship 해야 하나: 오늘의 병렬성은 모델이 분기를 skip해도 finalize에서야 발견됨. 이 통합 모델은 engine을 source of truth로 만든다 — 누가 언제 어디서 돌고 어떤 state를 쓰고 산출이 일치하는가까지. PR #230의 frozen failure_code 디시플린은 그대로 보존 (5→11), no_parallel_explanation 게이트도 그대로. Wall-clock 이득은 적당(1.4–1.8×)하지만, 진짜 페이로프는 invisible-failure mode 제거다.

한 줄 요약

엔진은 dispatcher 1개 (150 LOC) + 정책 모듈 ~10개 (각 200~400 LOC)로 명확히 쪼개진다. 47개 hook이 잡던 디테일은 8개 정책 모듈 + 2개 관찰 모듈 + 2개 상태 모듈에 추적 가능하게 흡수된다 (§3 매핑 표). 결정이 한 곳에서 일관되게 내려지면서, "어느 hook이 충돌하는지" 류의 현재 디버깅 비용이 사라지고, 새 정책 90%는 YAML 한 줄로 추가된다.
